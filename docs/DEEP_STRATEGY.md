## 4. THE INVISIBLE UX: GEO-LANGUAGE ROUTING

### Middo's Principle

> "Seniors should not even see the site as multi-language. They usually don't speak multiple languages. The change should happen in the footer only."
>
> "I send a link to my dad, page opens, it detects he is French. I send the same link to my Spanish grandmother, the same link opens the Spanish version."

This is the killer UX insight. One URL, automatic language. Zero cognitive load for the senior.

### Technical Implementation: Cloudflare Pages Middleware

Cloudflare Pages supports `functions/_middleware.js` that runs BEFORE every request. Cloudflare provides `request.cf.country` automatically (free, no API needed).

```
Visitor from France â†’ facil.guide/guides/make-phone-call/
  â†’ Middleware reads cf.country = "FR"
  â†’ Internally rewrites to /fr/passer-appel-iphone/ content
  â†’ URL stays facil.guide/guides/make-phone-call/ (or redirects to /fr/ version)
```

### Routing Logic

**Option A: Smart Redirect (Recommended)**

1. Visitor hits `facil.guide/guides/{en-slug}/`
2. Middleware checks `request.cf.country`
3. Maps country to language: FRâ†’fr, ES/MX/AR/COâ†’es, PT/BRâ†’pt, ITâ†’it, everything elseâ†’en
4. 302 redirects to `facil.guide/{lang}/{localized-slug}/`
5. Cookie set: `facil_lang=fr` â€” subsequent visits respect cookie over IP
6. Footer has language selector (tiny, unobtrusive) to override

**Why 302 not 301**: The same URL should be indexable in EN for Google. The redirect is for humans, not bots. Googlebot crawls from US â†’ gets EN version. French user â†’ gets FR version.

**Option B: Accept-Language Header (Supplement)**

Also check `Accept-Language` header as fallback when country is ambiguous:
- Swiss visitor (CH) could be FR/DE/IT â†’ check Accept-Language
- Belgian visitor (BE) could be FR/NL â†’ check Accept-Language
- Canadian visitor (CA) could be EN/FR â†’ check Accept-Language

**Option C: Universal URLs with Content Negotiation**

Most advanced: one URL (`/guides/make-phone-call/`) serves different HTML based on detected language. Same URL, different content. Strongest for sharing ("send the same link").

Technical: Middleware fetches the appropriate static page and serves it. `Vary: Accept-Language` header for caching.

### The "Send Link" Flow (Middo's Use Case)


For Google, we ALSO need language-specific URLs (`/fr/passer-appel-iphone/`) with hreflang tags for proper multilingual indexing. Solution:

- **Universal URL** (`/guides/make-phone-call/`): For sharing. Content-negotiated. Canonical points to language-specific version.

## 5. COMMUNITY & USER BASE MODEL

### The Problem With "Community" for Seniors

Seniors won't create accounts, join forums, or earn badges. That's for younger people. But helpers WILL. And institutions WILL.

### Three-Tier User Model

#### Tier 1: Anonymous Visitor (The Senior)
- No account needed. Ever.
- Reads guides, follows steps, bookmarks pages
- Can print/download PDF
- Can share via "Send to family" button
- This is 95% of traffic and must remain 100% frictionless

#### Tier 2: Helper Account (The Adult Child) â€” FREE
- Optional sign-up (email or Google)
- **Guide Library**: Save favorite guides, organize by family member ("Mom's iPhone guides", "Dad's Android guides")
- **Send & Track**: Send guides to parents, see if they opened them
- **Request a Guide**: "I need a guide for [specific thing]" â†’ goes into our content queue
- **Contribute Translations**: Native speakers can submit translation improvements
- **Progress Tracking**: Mark which guides their parent has completed
- **Notifications**: "New guide available in your parent's language"

This creates a USER BASE without making seniors create accounts. The helpers are the users.

#### Tier 3: Contributor / Institution â€” FREE or B2B
- **Individual Contributors**: Write/translate guides, earn reputation
- **Institutions**: Libraries, telecoms, elder care â€” white-label access, bulk downloads, embed widgets
- **Government Programs**: Official recommendation, co-branding

### Point System / Gamification (Helpers + Contributors)

**"Facil Points"** â€” earned by:
- Sending a guide to a parent: 5 points
- Parent opens the guide: 10 points
- Submitting a guide request: 5 points
- Contributing a translation fix: 25 points
- Writing a new guide (approved): 100 points
- Referring another helper: 50 points

**Leaderboard**: "Top Helpers This Month" â€” public recognition
**Badges**:
- "First Guide Sent" (onboarding)
- "Multilingual Helper" (sent guides in 2+ languages)
- "Guide Author" (contributed a guide)
- "Translation Hero" (fixed 10+ translations)
- "Family Tech Support" (sent 20+ guides)

**Why this works**: Helpers feel good about helping. Points quantify that. It's not about rewards â€” it's about recognition. "I've helped my mom complete 15 guides" is genuinely satisfying.

### Content Contribution Model

Like Wikipedia but curated:
1. Anyone can submit a guide or translation improvement
2. AI-assisted review checks quality, accuracy, and language
3. Approved submissions go live with contributor credit
4. Top contributors get a public profile: "Maria from Lisbon â€” 47 guides translated"

This scales content production beyond what we can do alone. If 100 helpers each fix one translation, that's 100 translations we didn't have to write.

### The "Facil Ambassador" Program

Recruit helpers who are particularly active:
- They become local ambassadors
- They know what their parents/community needs
- They suggest guides based on real struggles
- They test guides with real seniors and report back
- In return: early access to new features, ambassador badge, quarterly "thank you" newsletter

## 6. BEATING WIKIHOW

### Why WikiHow Is Vulnerable

1. **Traffic collapsed 85%** (300M â†’ 45M) since 2020. AI answers replaced their generic "how to" content.
2. **Ads everywhere**. Seniors hate popups. WikiHow has aggressive ad placement.
3. **Not senior-focused**. "How to make a phone call on iPhone" on WikiHow is written for everyone. It assumes tech literacy.
4. **Translations are auto-generated garbage**. WikiHow's non-English content is machine-translated and reads terribly.
5. **No mobile-first for seniors**. Their responsive design is fine for 25-year-olds, not for 75-year-olds with arthritis.

### How We Beat Them

**Strategy: don't fight the gorilla. Be the gorilla in rooms they can't enter.**

The combination of: zero competition + 5 languages + high intent + high volume + no ads (user stays) = compounding organic growth.
